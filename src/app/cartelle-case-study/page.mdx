# 🛒 Cartelle - Custom eCommerce Platform

A fully custom ecommerce platform built with Next.js (App Router), Prisma, and Stripe, focused on inventory safety and clear server architecture.

## 🔗 Live Links

-   **Storefront** - https://cartelle-mock-store.vercel.app
-   **Admin** - https://cartelle-mock-store.vercel.app/admin
-   **Storybook** - https://cartelle-mock-store-storybook.vercel.app/

## 🎯 Goal

Build a production-style ecommerce architecture implementing atomic inventory reservation, Stripe checkout, and admin tooling using a modern full-stack React architecture.

## 🏗️ Architecture Overview

Built with Next.js (App Router) using Server Components and Server Actions for data mutations. Prisma + PostgreSQL handle persistence, and Stripe manages payment intents and webhooks.

## 🧩 Key Engineering Decisions

-   Checkout-time inventory reservation to prevent overselling.
-   Clear boundary between Server Actions and API Routes.
-   Container/presentational component split for Storybook isolation.

## 🧠 System Design Deep Dive

### 1️⃣ Inventory Reservation

**Problem:** Multiple users can enter the checkout flow with the same item in their cart. If their combined quantity exceeds available stock, both users could complete the entire checkout flow and trigger the Stripe webhook before the system detects the oversell.

**Alternatives considered:**

1. Check stock at payment confirmation: Validate when the Stripe webhook fires, but this means users may not discover stock issues until after entering all payment details.
2. Reserve stock at add to cart: Temporarily hold the item for a user when they add it to their cart. Avoided as stock would be locked by people who are just browsing and never intend to buy.

**Solution:** Reserve stock when the checkout button is clicked. Once a user initiates checkout, the conversion intent is higher at this stage, so stock is reserved only for users who are more likely to complete their purchase. To handle abandoned checkouts, we set up a cron job using an API route and `vercel.json` to move carts from `CHECKOUT` to `ABANDONED` after 15 minutes of inactivity.

**Trade-offs:** Being on a Vercel Hobby plan meant that the cron job could run only once a day. In a production system this would be replaced with a more frequent job or a Redis-based time to live on the reservation key, which handles expiry atomically at the data layer.

### 2️⃣ Server Actions vs API Routes

**Problem:** Next.js supports both Server Actions and API Routes for handling server-side logic, but using them interchangeably without a clear convention leads to inconsistency. The question was: when should each be used, and why?

**Alternatives considered:**

1. Use API routes for everything: The traditional Next.js approach but introduces unnecessary HTTP overhead for simple data mutations that don't need to be externally accessible.
2. Use Server Actions for everything: Removes HTTP overhead entirely but Server Actions cannot receive requests from external services or handle certain browser APIs.

**Solution:** Use Server Actions as the default for most data mutations since they run directly on the server without a dedicated HTTP endpoint, reducing overhead. However, API routes are used in three specific cases where Server Actions are not viable:

1. When passing large files. Server Actions have a payload size limit, so sending large binary files through them will fail or be rejected.
2. When an external service needs to call your endpoint, such as a Stripe webhook or CRON job from `vercel.json`. Server Actions can only be invoked by your own application.

**Trade-offs:** Relying primarily on Server Actions makes debugging more challenging, as there is no clear request/response cycle that API routes expose in the network tab.

### 3️⃣ Component Splitting

**Problem:** Components that mix server-side logic with JSX are difficult to test and preview in isolation. A component tightly coupled to Server Actions or data fetching cannot be rendered without its full execution context.

**Alternatives considered:**

1. Keep logic and UI together: Simpler file structure but makes isolated testing and visual storybook previewing impossible without mocking the entire server context.
2. Mock server context in Storybook: Adds complexity to the testing setup that breaks when server logic changes.

**Solution:** Split components into two layers following the container/presentational pattern:

-   `CheckoutButton` - contains the Server Action and logic
-   `CheckoutButtonUI` - receives everything as props and is pure JSX

This means the UI layer can be rendered in any context, including Storybook, without needing a server environment.

**Trade-offs:** This approach introduces more files and prop drilling. It also complicates barrel exporting, as exporting server components and client components from the same index.ts can cause conflicts in Storybook.

## ✨ Features

-   🛒 **Persistent cart with server-side mutations** - cookie-backed cart ID, quantity guardrails, and transactional updates for line items.
-   💳 **Stripe checkout with webhook-driven order lifecycle** - payment intent creation, multi-step checkout UI, and webhook-based order state transitions.
-   📦 **Checkout-time inventory reservation** - stock is locked at checkout, released on timeout, and finalized on successful payment.
-   ⚙️ **Admin CMS for products and blog** - create/edit tooling for products, authors, and content with role-friendly flows.
-   📚 **Storybook-driven component system** - isolated UI previews, reusable presentational components, and faster iteration.
